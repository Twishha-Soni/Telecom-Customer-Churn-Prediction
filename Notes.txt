--- Steps in preprocessing data before Training the model ---

1. Basic exploration of the dataset like tracking the data types of the features available.
2. Stratified train/test split — stratify directly on the Churn label itself and keep the test set away for evalutaion at the end.
3. Full visualization pass — look for correlations, combined/derived attributes worth engineering.
4. Separate features (X) from label (y) in train set.
5. Clean data using SimpleImputer.
6. Handle categorical/text columns with OneHotEncoder.
7. Feature scaling — standard scaling for most numeric features, but for any
   heavy long-tailed distribution (e.g. TotalCharges, MonthlyCharges if
   skewed), consider an RBF-kernel similarity transform instead of plain
   scaling or simple log transformation of the heavy long tail data.
8. Custom transformers where useful (FunctionTransformer / a custom class with
   fit/transform)
9. Combine all of the above into a single sklearn Pipeline
   (ColumnTransformer + custom transformers)
10. Select 3 candidate models, train each, evaluate on a validation set (not
    test set yet) — this is a first-pass comparison, not final tuning.
11. Then tune your model using hyperparameter tuning for the model you think that would predict more accurate on the data.



# Step2:
-  If you evaluate on data the model was trained on (or worse, if you peek at the test set while engineering features), your performance numbers become optimistic — you're measuring memorization, not generalization.

- why stratified?
  With a ~73/27 class imbalance, a plain random split can — just by chance — shift that ratio a bit in either the train or test set, especially since 27% minority class isn't huge to begin with.

-  Stratifying on Churn guarantees both splits preserve the same ~73/27 ratio, so any performance differences you observe later are due to the model/features — not an artifact of a lucky or unlucky split.

# step3:
- you only explore train_set from this point forward, never test_set.

- The reasoning: anything you learn from visualizing data — a correlation you notice, an outlier pattern, an idea for a combined attribute — ends up influencing decisions you make about feature engineering. If those decisions were shaped by patterns you saw in the test set, you've leaked information from the "future" (data you're supposed to be blind to) into your model design.

# step4:
- the imputer must be fit only on training data, then that same fitted transformation (e.g., "fill missing values with 32.5") gets applied to the test set and to any future data at serving time. If you fit on the full dataset instead, statistics from the test set (or unseen future customers) leak into your training process

- this string→numeric conversion isn't something SimpleImputer does — it's a separate fix that has to happen before imputation, and it will eventually need to live inside your actual pipeline (not just as a manual one-off here), so that raw incoming data at serving time gets the same treatment. 

- TotalCharges's missing values aren't random — every blank corresponds to a customer with tenure == 0 (they haven't been billed yet). Domain knowledge says the correct fill value for those rows isn't "whatever the median happens to be" — it's 0, since a brand-new customer genuinely has $0 in total charges.
